## 1. 目的と背景

本プロジェクトは、3名のチームによるJava学習を主目的として、TCPソケット上で動作するマルチスレッドチャットツールを開発するものである。プログラミング経験者（5年以上）がJavaを習得する題材として、標準ライブラリのみで完結し、並行処理・ネットワークI/O・プロトコル設計・チーム開発の実践を一通り経験できることを狙いとする。

成果物そのものの実用性は目的としない。学習効果を最大化するため、既存のネットワークフレームワーク（Netty等）は使用せず、`java.net`・`java.io`・`java.util.concurrent` を直接扱う。

## 2. スコープ

### 2.1 対象とするもの

- TCP上の独自テキストプロトコルによるサーバ・クライアント間通信
- 複数クライアントの同時接続と、ルーム単位のメッセージ中継
- CLIクライアント
- プロトコル層の単体テストと、サーバの結合テスト

### 2.2 対象としないもの（初版では実装しない）

- 暗号化（TLS）・認証（パスワード）
- メッセージの永続化（履歴保存）
- GUIクライアント（拡張フェーズの候補とする）
- サーバの多重化・クラスタリング

## 3. 用語定義

| 用語 | 意味 |
|------|------|
| セッション | 1つのTCP接続に対応するサーバ側の管理単位 |
| 登録 | クライアントが `NICK` によりユーザー名を確定した状態 |
| ルーム | メッセージ配送のグループ。`JOIN` で参加、参加者全員にメッセージが中継される |

## 4. システム構成

サーバ1プロセスに対し、複数のクライアントプロセスがTCPで接続するスター型構成である。クライアント同士が直接通信することはない。

```
+----------+     TCP      +----------+
| Client A | <----------> |          |
+----------+              |          |
+----------+              |  Server  |
| Client B | <----------> |          |
+----------+              |          |
+----------+              |          |
| Client C | <----------> |          |
+----------+              +----------+
```

- 待受ポート: デフォルト `6060`（起動引数で変更可能）
- 文字コード: UTF-8 固定
- 1行 = 1メッセージ（LF終端）の行指向プロトコル

## 5. 機能要件

| ID | 要件 | フェーズ |
|----|------|----------|
| F-1 | クライアントはサーバに接続し、ユーザー名を登録できる | M1 |
| F-2 | ユーザー名はサーバ内で一意である。重複時はエラーを返す | M1 |
| F-3 | クライアントはルームに参加・退出できる。ルームは最初の参加時に自動生成され、無人になったら消滅する | M2 |
| F-4 | ルーム宛のメッセージは、そのルームの参加者全員（送信者を含む）に配送される | M1(単一ルーム)/M2(複数) |
| F-5 | ルーム一覧・ルーム内ユーザー一覧を取得できる | M2 |
| F-6 | 参加・退出は同ルームの他ユーザーに通知される | M2 |
| F-7 | クライアントの切断（正常・異常とも）を検知し、全ルームから退出させる | M3 |
| F-8 | 不正な入力（未知コマンド、長すぎる行、未登録での操作）にはエラー応答を返し、切断はしない | M3 |

## 6. 非機能要件

| ID | 要件 |
|----|------|
| N-1 | 同時接続50クライアントで正常動作すること |
| N-2 | 1行の最大長は4096バイトとし、超過した場合はエラー応答の上で当該行を破棄する |
| N-3 | 依存ライブラリはJava標準ライブラリのみとする（テストのJUnit 5を除く） |
| N-4 | 応答が遅いクライアントが1つ存在しても、他クライアントへの配送が停止しないこと（M3） |
| N-5 | サーバはCtrl-C（SIGINT）で全接続を閉じて正常終了すること |

## 7. プロトコル仕様

本章が3名の開発の**共通契約**である。変更する場合は必ず3名の合意を経て本書を改版する。

### 7.1 基本形式

- トランスポート: TCP
- エンコーディング: UTF-8
- フレーミング: LF (`\n`) 終端の1行を1メッセージとする。CRLFを受信した場合はCRを除去して解釈する
- 書式: `コマンド [引数1] [引数2] [本文...]`。コマンドと引数は半角スペース区切り。**最後の引数（本文）のみスペースを含んでよい**
- ユーザー名・ルーム名: `[A-Za-z0-9_-]{1,16}`

### 7.2 クライアント → サーバ

| コマンド | 書式 | 意味 | 前提状態 |
|----------|------|------|----------|
| NICK | `NICK <name>` | ユーザー名を登録する | 接続直後（未登録） |
| JOIN | `JOIN <room>` | ルームに参加する | 登録済み |
| LEAVE | `LEAVE <room>` | ルームから退出する | 参加中 |
| MSG | `MSG <room> <text>` | ルームへ発言する | 参加中 |
| ROOMS | `ROOMS` | ルーム一覧を要求する | 登録済み |
| USERS | `USERS <room>` | ルーム内ユーザー一覧を要求する | 登録済み |
| PING | `PING` | 生存確認 | 任意 |
| QUIT | `QUIT` | 切断を予告する | 任意 |

### 7.3 サーバ → クライアント

| 応答 | 書式 | 意味 |
|------|------|------|
| OK | `OK <command> [詳細]` | 直前のコマンドの成功応答。例: `OK JOIN general`、`OK ROOMS general dev` |
| ERR | `ERR <code> <理由>` | エラー応答（下表） |
| MSG | `MSG <room> <sender> <text>` | 発言の中継 |
| INFO | `INFO <JOIN\|LEAVE> <room> <name>` | 参加・退出の通知 |
| PONG | `PONG` | PINGへの応答 |
| BYE | `BYE` | サーバ側からの切断予告（QUIT受理・シャットダウン時） |

### 7.4 エラーコード

| code | 意味 |
|------|------|
| 400 | 書式不正（未知コマンド、引数不足、行長超過） |
| 401 | 未登録（NICK前に他コマンドを送った） |
| 403 | 操作不可（ユーザー名重複、未参加ルームへのMSG/LEAVE） |
| 404 | 対象なし（存在しないルームへのUSERS） |

### 7.5 状態遷移

セッションは3状態を持つ。

```
接続 --> [未登録] --NICK成功--> [登録済み] --QUIT/切断--> [終了]
            |                        |
            +--NICK以外: ERR 401     +--JOIN/LEAVE/MSG/...
```

### 7.6 通信例

```
C: NICK alice
S: OK NICK alice
C: JOIN general
S: OK JOIN general
S: INFO JOIN general alice      （※同ルームの全員に配送）
C: MSG general こんにちは
S: MSG general alice こんにちは （※alice自身にも配送される）
C: QUIT
S: BYE
```

## 8. サーバ設計（スレッドモデル）

段階的に2つのモデルを実装する。この移行自体が学習項目である。

**フェーズ1（M1〜M2）: 1接続1スレッド・直接書き込み**
accept専用スレッドが接続を受け、接続ごとにreaderスレッドを1本立てる。配送は送信先ソケットへ直接 `write` する（ソケット単位で `synchronized`）。共有状態（ユーザー表・ルーム表）は `ConcurrentHashMap` で管理する。

**フェーズ2（M3）: 送信キュー分離**
クライアントごとに送信用 `BlockingQueue` とwriterスレッドを持たせ、配送は「キューに積むだけ」にする。これにより応答の遅いクライアントが配送全体を止める問題（N-4）を解消する。キューが上限に達したクライアントは切断する。

## 9. モジュール構成と分担

Gradleマルチモジュール構成とする。

| モジュール | 内容 | 主担当 |
|------------|------|--------|
| `chat-protocol` | メッセージ型の定義、パース・整形（純粋ロジック、I/O非依存） | 全員で初週に実装し、以後C |
| `chat-server` | 接続受付、セッション管理、ルーム管理、配送 | A |
| `chat-client` | CLIクライアント（受信表示スレッド＋入力送信） | B |
| （横断） | 結合テスト、CI（GitHub Actions）、ビルド整備 | C |

`chat-protocol` をI/Oから分離するのは、単体テストを可能にするための必須の設計判断である。サーバ・クライアントとも、ソケットから読んだ行を `chat-protocol` のパーサに渡し、返ってきたメッセージ型で処理を分岐する。

## 10. リポジトリ構成（フォルダ構成）

Gradleの標準ディレクトリレイアウト（`src/main/java`・`src/test/java`）に従う。パッケージのルートは `chat` とする。

```
chat/
├── SPEC.md                     # 本書
├── README.md                   # ビルド・起動手順
├── settings.gradle.kts         # モジュール一覧の定義
├── build.gradle.kts            # 全モジュール共通のビルド設定
├── gradlew / gradlew.bat       # Gradle Wrapper（各自のGradleインストール不要）
├── gradle/wrapper/             # Wrapper本体
├── .github/
│   └── workflows/
│       └── ci.yml              # push/PR時に ./gradlew build を実行
├── chat-protocol/              # 【共通契約】メッセージ型とパーサ（I/O非依存）
│   ├── build.gradle.kts
│   └── src/
│       ├── main/java/chat/protocol/
│       │   ├── Message.java            # メッセージ型（sealed interface + record）
│       │   ├── MessageParser.java      # 1行 → Message
│       │   └── MessageFormatter.java   # Message → 1行
│       └── test/java/chat/protocol/
│           └── MessageParserTest.java
├── chat-server/                # 担当A（chat-protocolに依存）
│   ├── build.gradle.kts
│   └── src/
│       ├── main/java/chat/server/
│       │   ├── ChatServer.java         # main。accept ループ
│       │   ├── ClientSession.java      # 1接続の状態と reader スレッド
│       │   ├── RoomRegistry.java       # ルーム表（ConcurrentHashMap）
│       │   └── UserRegistry.java       # ユーザー表
│       └── test/java/chat/server/
│           └── ChatServerIT.java       # 実ソケットでの結合テスト（担当C）
└── chat-client/                # 担当B（chat-protocolに依存）
    ├── build.gradle.kts
    └── src/
        └── main/java/chat/client/
            ├── ChatClient.java         # main。接続と入力ループ
            └── ReceiverThread.java     # 受信・表示スレッド
```

クラス構成は起点の目安であり、実装の進行に応じて各担当の裁量で分割してよい。ただしモジュール間の依存方向（`chat-server` → `chat-protocol` ← `chat-client`、サーバとクライアントは互いに依存しない）は変更しない。

## 12. Git運用（ブランチ戦略）

本章は原則を定める。コマンドレベルの具体的な作業手順は [CONTRIBUTING.md](CONTRIBUTING.md) に定め、日々の作業ではそちらに従う。

GitHub Flowを採用する。常設ブランチは `main` のみとし、`develop` ブランチは置かない。3人規模・単一成果物の開発では、Git Flowのような多段ブランチは同期コストがメリットを上回るためである。

**基本ルール**

`main` は常にビルド可能・全テスト成功の状態を保つ。`main` への直接pushは禁止し（GitHubのブランチ保護で強制する）、すべての変更はPull Request経由とする。PRのマージ条件は、CI成功と、他メンバー1名以上のapproveの2点である。

**ブランチの切り方**

作業は必ず `main` から分岐する。命名は `種別/内容` とし、種別は次の3つに限定する。

| 種別 | 用途 | 例 |
|------|------|-----|
| `feature/` | 機能追加 | `feature/join-command`、`feature/client-receiver-thread` |
| `fix/` | 不具合修正 | `fix/nick-duplicate-check` |
| `docs/` | 文書のみの変更 | `docs/spec-protocol-v2` |

1ブランチ＝1トピックとし、PRの差分は目安300行以内に収める。大きな機能は「パーサ追加」「サーバ側処理」「クライアント側処理」のように分割してPRを重ねる。差分が小さいほどレビューが機能し、コンフリクトも起きにくい。

**マージとタグ**

マージ方式はsquash mergeに統一し、`main` の履歴を1PR＝1コミットに保つ。マージ済みブランチは削除する。マイルストーン完了時には `v0.1-m1` のようにタグを打ち、後から各段階の状態に戻れるようにする。

**コンフリクトの予防**

`chat-protocol` の変更（＝7章の改版）だけは3人の合意を必須とし、専用PRとして単独で出す。担当モジュールが分かれているため、これを守る限りコンフリクトはほぼ `chat-protocol` とビルド設定に限られる。作業開始時と PR作成前に `git pull --rebase origin main` で追従することを習慣とする。

## 14. テスト方針

`chat-protocol` はパーサ・整形の全コマンド・全エラー系をJUnitで網羅する（担当C、ただし各自が触った箇所のテストは各自が書く）。サーバは、テストコード内から実ソケットで複数クライアントを接続する結合テストを書き、「2クライアントが同一ルームで会話できる」「片方の異常切断後にもう片方が影響を受けない」を自動検証する。手動の目視確認のみで完了とするマイルストーンは設けない。
